Case Studies

Deep dives into two projects — the reasoning behind the architecture, what broke during experimentation, and what I'd do differently. Click a section to expand.

Agentic AI Developer Tools Research Agent +
Problem

Evaluating developer tools (frameworks, SaaS APIs, libraries) manually meant hours of tab-switching between docs, pricing pages, and GitHub — for every single comparison, repeated from scratch each time.

Constraints

Needed to run on a single machine with no dedicated scraping infrastructure, respect target sites' rate limits, and return a structured report in well under a minute per tool.

Architecture

A 3-node LangGraph state machine (Extract → Research → Analyze) with Pydantic schemas enforcing structured output at every hop, exposed behind a FastAPI service.

Experiments

Sequential extraction was the first version and took ~40s/tool. Introducing a ThreadPoolExecutor for parallel fan-out across sources plus an LRU cache for repeated domains cut that dramatically without changing output quality.

Results

75% reduction in end-to-end research latency, with deterministic, schema-validated output ready to render directly into a comparison report.

Lessons Learned

Caching keyed too aggressively at first (whole-query level) and produced stale results — moving the cache key to the per-source domain level fixed correctness while keeping the speedup.

LLM Fine-Tuning Phi-2 Skincare Domain Advisor +
Problem

General-purpose LLMs give vague, non-committal answers about skincare ingredient safety and interactions — not useful for a focused advisory tool.

Constraints

Only 4GB of consumer VRAM available — full fine-tuning of a 2.7B model was out of the question, so the entire approach had to fit a parameter-efficient method.

Architecture

QLoRA over Microsoft Phi-2 with 4-bit quantization (BitsAndBytes), LoRA adapters via PEFT/TRL, and an Alpaca-style instruction dataset curated for ingredient-safety Q&A.

Experiments

Iterated on LoRA rank and target modules to balance trainable-parameter count against answer quality, alongside gradient checkpointing to keep peak memory under budget.

Results

Only 1–2% of parameters trainable, full fine-tune completed inside 4GB VRAM, with noticeably more grounded, domain-specific answers than the base model.

Lessons Learned

Dataset quality mattered more than hyperparameter tuning — cleaning up ambiguous or contradictory instruction examples gave a bigger quality jump than any LoRA rank change.

AI Engineer — Agentic Retrieval-Augmented Generation Platform
  • Designed and built a production-grade agentic RAG orchestration system as a LangGraph state machine, coordinating query analysis, multi-source retrieval, relevance grading, and hallucination-grounded generation into a single reliable pipeline exposed via FastAPI.
  • Engineered a parallel multi-source retrieval layer spanning vector search, web search, and structured SQL/graph queries, merging results through deduplication and cross-encoder reranking to reduce end-to-end latency.
  • Implemented bounded self-correction loops — capped relevance re-retrieval and hallucination-regeneration cycles with hard iteration limits — ensuring the system always terminates deterministically and returns a grounded answer, even after repeated grader or model failures.
  • Applied fail-open reliability design across grading and reranking components (automatic fallback to passthrough scoring, conservative defaults on parse failures) so malformed model output never collapses retrieval context to empty.
  • Added a state-reset reducer to prevent stale retrieval context from bleeding across retry cycles, and per-thread conversation isolation via a persistence checkpointer to safely support concurrent multi-user sessions, with graceful degradation when persistence is unavailable.
  • Optimized for heavy concurrent load through parallel retrieval fan-out and stateless-fallback API design, prioritizing predictable behavior over best-case-only performance.
